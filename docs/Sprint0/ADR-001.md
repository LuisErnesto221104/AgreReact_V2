## 1. Contexto y Problema

AgroApp debe operar en zonas rurales de Zitácuaro y Ocampo, Michoacán, donde la conectividad a internet es nula o muy limitada. El sistema necesita:

- Persistir datos de ganado (animales, eventos sanitarios, gastos, historial)
- Emitir notificaciones locales programadas (AlarmManager)
- Generar reportes PDF
- Todo esto **sin ninguna dependencia de red en tiempo de ejecución**

La versión v1.0 fue concebida como aplicación Android nativa en Java. La **SCR-001** (aprobada Marzo 2026) propone migrar la capa de interfaz de usuario a React Native, manteniendo el backend Java y SQLite ya probados.

---

## 2. Factores de Decisión

- **Offline-First absoluto**: sin INTERNET permission en `AndroidManifest.xml` (la "prueba reina")
- **Backend Java existente**: `DatabaseHelper.java` v4, 6 tablas, 6 DAOs ya implementados
- **ACID requerido**: `RNF001` — escrituras síncronas, sin pérdida de datos ante cierres forzados
- **Equipo mixto**: un desarrollador frontend más fluido en JavaScript/React
- **Restricción de tiempo académico**: 5 sprints, 2 meses

## 4. Contrato de Asincronía JS↔Java

Toda llamada de React Native a un Native Module es **asíncrona y basada en Promises**. El hilo JavaScript **nunca bloquea esperando** una operación de base de datos.

```typescript
// TypeScript (src/native/AnimalModule.ts)
interface AnimalModule {
  insertarAnimal(data: AnimalData): Promise<number>;
  obtenerAnimales(): Promise<Animal[]>;
  eliminarAnimal(arete: string): Promise<boolean>;
}
```

```java
// Java (android/.../modules/AnimalModule.java)
@ReactMethod
public void insertarAnimal(ReadableMap data, Promise promise) {
    executorService.execute(() -> {
        try {
            long id = animalDAO.insertAnimal(data);
            promise.resolve((double) id);
        } catch (Exception e) {
            promise.reject("DB_ERROR", e.getMessage());
        }
    });
}
```

**Reglas del contrato:**
- Siempre `ExecutorService` para operaciones de BD (nunca en el hilo JS)
- `AsyncTask` está **prohibido** — incompatible con `targetSdk 34+`
- Nunca serializar JSON crudo — usar `ReadableMap` / `ReadableArray`
- Siempre resolver o rechazar el Promise — nunca dejarlo pendiente

## 6. SQLite como Fuente Única de Verdad

- **No hay caché en memoria** que pueda quedar fuera de sync
- **No hay estado global en JS** para datos persistentes
- Toda lectura va directo a SQLite vía DAO
- Toda escritura es transaccional (ACID) — ver [`docs/DATABASE.md`](DATABASE.md)
- La UI se actualiza solo con datos confirmados de la BD

---

## 7. Arquitectura de Seguridad de Sesión

- Autenticación **local offline** — PIN/NIP 4–6 dígitos numéricos (RF013)
- El PIN se almacena hasheado en la tabla `usuarios` de SQLite
- Sin tokens de red, sin Firebase Auth, sin sesiones compartidas
- Un solo usuario por dispositivo (sin multiusuario)
- Acceso por rol: `ADMIN` (configuración) / `USUARIO` (operación)

---

## 8. Consecuencias Positivas

- Backend Java probado y estable reutilizado sin cambios
- React Native permite UI más rápida de desarrollar para Jovani (frontend)
- Hermes Engine incluido en RN 0.84.1 = rendimiento optimizado sin configuración extra
- Offline-first garantizado por diseño (no por configuración)
- SQLite ACID = datos seguros ante cierres inesperados

## 9. Consecuencias Negativas / Riesgos

| Riesgo | Nivel | Mitigación |
|---|---|---|
| Complejidad del Bridge (@ReactMethod) | Medio | Guía técnica [`docs/BRIDGE.md`](BRIDGE.md) + SCR-001 |
| curva de aprendizaje Bridge (@ReactMethod) | Medio | Spike en Sprint 0 (SCR-001) |
| Debugging entre dos lenguajes (JS + Java) | Medio | Logging en ambas capas, Flipper |
| `targetSdk 36` puede deprecar APIs Java | Bajo | NDK 27.1.12297006 estable en API 36 |

## 10. Alternativas Rechazadas

### Android Nativo Puro (Java)
**Rechazado:** Desarrollo de UI más lento, Jovani (frontend) sin experiencia previa en XML/Views de Android. El tiempo académico no permite la curva de aprendizaje.

### React Native con Expo
**Rechazado:** Expo no permite acceso directo a `SQLiteOpenHelper` nativo ni a `AlarmManager` exacto. El sistema requiere control total sobre la BD y notificaciones locales.

### Flutter
**Rechazado:** El backend Java ya está implementado. Migrar a Dart implicaría reescribir toda la capa de datos. Sin ventaja técnica que justifique el costo.

### Firebase / Room con sincronización
**Rechazado:** Viola el requisito offline-first absoluto. Firebase requiere internet para autenticación y sync. Room añade una capa de abstracción innecesaria sobre una BD SQLite ya probada.

### React Native con SQLite vía librería JS (expo-sqlite)
**Rechazado:** No permite usar el `DatabaseHelper.java` existente con sus 6 DAOs y la lógica de AlarmManager integrada. Las librerías JS-SQLite tienen limitaciones en transacciones ACID complejas.
